# 🚀 第一章 计算机系统概述

## 第一模块：硬件“四大金刚”与寄存器 (填空/名词解释区)

计算机底层的物理世界由四个核心部件组成，它们构成了操作系统的主要管理对象：

1. ***\*处理器 (Processor/CPU)\****：控制计算机操作，执行数据处理功能。
2. ***\*主存储器 (Main Memory)\****：也叫实存储器。注意它的核心考点：***\*易失性 (Volatile)\****，一旦关机内容就丢失了。
3. ***\*输入/输出模块 (I/O Modules)\****：负责在计算机和外部环境（如硬盘、终端、通讯设备）之间移动数据。
4. ***\*系统总线 (System Bus)\****：处理器、内存和 I/O 模块之间通信的高速公路。

***\*🌟\**** ***\*重点辨析：寄存器的两副面孔\****

处理器内部有两类非常关键的寄存器，考试常考分类：

- ***\*用户可见寄存器\****：程序员可以用的（如数据寄存器、地址寄存器、段指针、栈指针），用来优化内存引用。
- ***\*控制和状态寄存器（OS的核心武器）\****：对普通用户不可见，用于控制程序执行。必背的三个：
  - ***\*PC (程序计数器)\****：永远保存***\*下一次要取的指令地址\****。
  - ***\*IR (指令寄存器)\****：保存***\*当前最近获取到的指令\****。
  - ***\*PSW (程序状态字)\****：包含条件码、中断允许/禁止位、内核/用户模式位。
  - **避坑提醒**：***\*条件码（如结果为正、负、零、溢出）是由硬件操作自动设置的，不能通过程序显式修改！\****

## 第二模块：核心机制：指令周期与中断 (简答题/逻辑推演)

***\*1. 基本的指令周期\****

处理器执行程序就干两步死循环：***\*取指令 (Fetch) $\rightarrow$ 执行指令 (Execute)\****。

每取完一次指令，PC 的值就会自动递增，指向下一条指令。

***\*2. 中断机制 (Interrupts) —— 为什么需要中断？\****

- ***\*痛点\****：I/O 设备（如打印机、硬盘）的速度比 CPU 慢得多。如果让 CPU 干等 I/O 完成，是对 CPU 资源的极大浪费。
- ***\*解决方案\****：引入中断。CPU 给 I/O 发完指令后就去干别的事，I/O 干完活了再“中断” CPU 报告结果。
- ***\*四大中断类型（常考选择题）\****：程序中断（如算术溢出、除以零）、定时器中断、I/O 中断、硬件故障中断。

***\*3. 发生中断时，CPU 是怎么处理的？（底层逻辑）\****

- 硬件层面，处理器会把当前的状态（***\*PSW 和 PC\****）压入***\*控制栈\****中保护起来，然后加载中断处理程序的入口地址到 PC 中。
- 等中断处理完，再从栈里把旧的 PSW 和 PC 弹出来，恢复原程序的执行。

***\*4. 多个中断“大撞车”怎么办？\****

如果正在处理一个中断时，又来了一个中断（比如边接收数据边打印），有两种处理策略：

1. ***\*顺序中断处理\****：处理一个中断时，**禁止（屏蔽）**其他中断发生，后来的只能排队。
2. ***\*嵌套中断处理\****：定义***\*中断优先级\****。优先级高的中断可以强行打断优先级低的中断。

## 第三模块：必考大题区：存储器层次结构与 Cache

***\*1. 存储器的“金字塔”层次结构\****

为了解决容量、速度和价格之间的矛盾，存储器被设计成金字塔结构（寄存器 $\rightarrow$ Cache $\rightarrow$ 主存 $\rightarrow$ 磁盘 $\rightarrow$ 磁带）。沿着金字塔往下看，规律如下（***\*必背\****）：

- ***\*每“位”的价格递减\****（越来越便宜）。
- ***\*容量递增\****（越来越大）。
- ***\*存取时间递增\****（越来越慢）。
- ***\*处理器访问存储器的频率递减\****（去底层找数据的次数越来越少）。

***\*2.\**** ***\*🌟\**** ***\*局部性原理 (Locality) —— 现代体系结构的灵魂\****

- ***\*空间局部性\****：处理器访问内存倾向于“簇化”（比如遍历数组，访问了地址 A，大概率马上会访问地址 A+1）。
- ***\*时间局部性\****：处理器倾向于访问最近刚使用过的数据（比如循环体中的变量）。

***\*3. 高速缓存 (Cache)\****

- ***\*原理\****：利用局部性原理，在 CPU 和主存之间放一个“小而快”的 Cache，保存一部分主存数据的副本。***\*Cache 对操作系统是不可见的\****。
- ***\*高速缓存设计的五大核心问题（常考填空或选择）\****：
  1. ***\*高速缓存大小\****。
  2. ***\*块大小\****（与内存交换的数据单位）。
  3. ***\*映射函数\****（决定主存块放进 Cache 的哪个位置）。
  4. ***\*置换算法\****（Cache 满了该踢谁出去？最常用的是 ***\*LRU 最近最少使用算法\****，淘汰最长时间未被访问的块）。
  5. ***\*写策略\****（更新数据时，是每次都写回主存，还是等块被替换出 Cache 时才写回？这关乎性能和数据一致性）。

## 第四模块：I/O 技术的“三次进化” (选择题大户)

计算机的发展史，就是 CPU 不断“甩锅” I/O 任务的历史。考卷上特别喜欢考这三种技术的优缺点对比：

1. ***\*可编程 I/O (Programmed I/O)\****：
   - **特点**：最原始。CPU 发出命令后，需要***\*周期性地不断检查\**** I/O 设备的状态（这叫“忙等待”）。
   - **致命缺点**：严重浪费 CPU 时间，导致整个系统性能严重下降。
2. ***\*中断驱动 I/O (Interrupt-Driven I/O)\****：
   - **特点**：CPU 发出命令后继续做其他事，I/O 准备好了再发中断信号通知 CPU。
   - **缺陷**：每次只能传输一个字的数据，而且***\*读写的每个字依然要经过 CPU\****。CPU 还是被大量的数据搬运工作给拖累了。
3. ***\*🌟\**** ***\*直接内存存取 (DMA, Direct Memory Access)\****：
   - **特点**：终极杀器。由一块专门的 DMA 芯片接管总线。
   - **优势**：将***\*整个数据块\****直接在 I/O 和内存之间传输，***\*完全不需要经过处理器\****！CPU 只在传输开始和结束时介入一下。
   - **代价**：在 DMA 大量传输数据占用总线时，CPU 可能会因为抢不到总线而稍微执行得慢一点（周期窃取），但这比前两种方法有效得多！

# 🚀 第二章 操作系统概述

## 第一模块：操作系统的“三大目标”与“两副面孔” (必考填空/简答)

首先，我们要搞清楚操作系统（OS）到底是个什么东西。

- ***\*三大主要目标\****：
  1. ***\*方便\****：作为用户和计算机之间的接口（让你不用手敲二进制代码）。
  2. ***\*有效\****：作为资源管理器（合理分配CPU和内存）。
  3. ***\*扩展能力\****：允许系统随着时间推移去升级硬件、添加新服务和纠正错误。
- ***\*两副面孔（核心定位）\****：
  - ***\*作为接口\****：它为程序开发、程序运行、I/O设备访问、文件访问控制、错误检测等提供服务。
  - ***\*作为资源管理器\****：操作系统本质上***\*也是一段程序\****，它靠处理器执行。它会经常把处理器的控制权交给其他程序，然后再想办法把控制权“抢”回来。

------

## 第二模块：操作系统的“进化史” (选择/排序题高频区)

操作系统不是一天建成的，它经历了四个极其重要的进化阶段。你必须记住它们的顺序和解决的痛点：

1. ***\*串行处理 (Serial Processing)\****：
   - **时代背景**：最早期的计算机（如1946年的ENIAC），***\*没有操作系统\****。
   - **痛点**：程序员直接操作硬件，全靠手工预约时间，准备时间极长，CPU大量时间被浪费在等待人工操作上。
2. ***\*简单批处理系统 (Simple Batch Systems)\****：
   - **核心发明**：***\*监控程序 (Monitor)\**** 的诞生。用户不再直接访问机器，而是把作业交给操作员打包（批处理），监控程序会自动加载下一个程序，减少了人工干预。
   - **硬件支持**：为了不让用户程序捣乱，硬件引入了***\*内存保护、定时器、特权指令和中断\****。
   - **考点（运行模式）**：由此诞生了***\*用户模式（User Mode，用户程序运行，权限低）和内核模式（Kernel Mode，监控程序运行，权限高）\****。
3. ***\*多道批处理系统 (Multiprogrammed Batch Systems)\****：
   - **痛点**：简单批处理时，由于I/O设备太慢，CPU经常处于空闲（闲着没事干）状态。
   - **伟大飞跃**：***\*多道程序设计 (Multiprogramming)\****！内存里同时装好几个程序。当程序A等待I/O时，CPU立刻切换去执行程序B。这极大提高了CPU和内存的利用率以及系统吞吐量。
4. ***\*分时系统 (Time Sharing Systems)\****：
   - **痛点**：批处理系统不能让人机交互。
   - **解决方案**：让处理器的时间在多个用户之间快速“切片”共享。最著名的早期系统是MIT开发的***\*CTSS（兼容分时系统）\****，它靠时钟中断（约0.2秒一次）来抢占处理器并切换用户。
   - **对比考点**：批处理的目标是***\*最大化处理器利用率\****，而分时系统的目标是***\*最小化响应时间\****。

------

## 第三模块：现代操作系统的“五大理论基石” (核心大题区)

操作系统发展至今，拿下了5个最重要的理论成就，这是重中之重：

***\*1. 进程 (Process)\****

- ***\*定义（必背）\****：一个正在执行的程序 / 一个单一顺序线程、当前状态和一组系统资源的活动单元。
- ***\*组成部分\****：一段可执行程序 + 相关数据 + ***\*执行上下文（进程状态，最关键！）\****。
- ***\*解决的问题\****：解决了并发执行时容易出现的错误，如不正确的同步、失败的互斥、死锁等。

***\*2. 内存管理 (Memory Management)\****

- ***\*五大职责\****：进程隔离、自动分配和管理、支持模块化程序设计、保护和访问控制、长期存储。
- ***\*虚存 (Virtual Memory) 与分页\****：允许程序以逻辑方式访问内存，物理上把程序切成固定大小的块（称为***\*页\****），实现动态映射，彻底打破了物理内存大小的限制。

***\*3. 信息保护和安全\****

- 记住四个关键词：***\*可用性\****（防中断）、***\*保密性\****（防偷看）、***\*数据完整性\****（防篡改）、***\*认证\****（验明正身）。

***\*4. 调度和资源管理\****

- 策略必须考虑三个要素：***\*公平性、有差别的响应性、有效性\****。

***\*5. 系统结构\****

- 将复杂的操作系统划分为层次结构，每一层只依赖下一层。

------

## 第四模块：现代操作系统的“新式武器” (概念连线题)

为了适应多核和网络时代，现代操作系统引入了以下新架构：

- ***\*微内核 (Microkernel)\****：把大部分功能踢出内核，内核只保留最基本的（如地址空间、IPC、基本调度）。优点是简化设计、极其灵活、***\*非常适合分布式环境\****。
- ***\*多线程 (Multithreading)\****：把进程进一步切分为***\*线程\****（可分派的工作单元）。进程成了资源分配的容器，线程才是真正在CPU上跑的实体。
- ***\*对称多处理 (SMP)\****：多个处理器共享内存和I/O，都能执行相同功能。优点是***\*性能高、可用性强（坏一个不至于全死）、可扩展\****。
- ***\*面向对象设计 & 客户-服务器模型\****：通过RPC（远程过程调用）通信，极大提高了操作系统的模块化和分布式计算能力。

## 第五模块：实战演练——Windows 与 UNIX/Linux 架构

考试中通常会以常识题出现，了解它们的设计特点：

- ***\*Windows体系结构\****：
  - 广泛使用了面向对象概念（封装、继承、多态）和客户-服务器模型。
  - ***\*内核模式\****包含：执行体（核心服务）、内核、HAL（硬件抽象层，屏蔽底层硬件差异）、设备驱动等。
  - ***\*用户模式\****包含：环境子系统、用户应用程序、服务进程等。
- ***\*UNIX/Linux体系结构\****：
  - **UNIX里程碑**：当年用 ***\*C语言重写\**** 是一个巨大飞跃。
  - **Linux特点**：由Linus Torvalds开发，免费开源。最核心的考点是它是***\*高度模块化的单体内核\****，支持***\*可加载模块\****（动态链接、可堆叠），不需要重新编译整个内核就能添加新功能。
  - **系统调用 (System Call)**：记住，所有用户应用程序想请求内核服务，都必须通过***\*系统调用接口\****。

# 🚀 第三章 进程描述和控制

## 第一模块：揭开“进程”的真面目 (必考概念/填空区)

很多零基础同学搞不懂程序和进程的区别。记住讲师一句话：***\*程序是死在硬盘上的代码，进程是活在内存里干活的实体！\****

***\*1. 进程的两大基本要素\****

当处理器开始执行代码时，这个执行实体就叫进程，它由两部分组成：***\*程序代码\**** 和 ***\*与代码相关联的数据集\****。

***\*2.\**** ***\*🌟\**** ***\*绝对核心：进程映像 (Process Image)\****

考试如果问“进程到底包含了什么物理结构”，一定要答这四个词：

- ***\*用户程序 (User Program)\****：你要执行的代码。
- ***\*用户数据 (User Data)\****：程序运行时要用的数据，属于用户空间的可修改部分。
- ***\*系统栈 (System Stack)\****：用来保存过程调用和系统调用的参数及返回地址。
- ***\*进程控制块 (PCB - Process Control Block)\****：***\*这是全章第一大重点！\****。

***\*3. 进程的“身份证与病历本”：PCB\****

- ***\*地位\****：PCB 是操作系统中***\*最重要的数据结构\****！操作系统就是靠它来管理进程的。
- ***\*里面装了什么？\**** (背诵口诀：***\*我是谁、我在哪、我有什么特权\****)
  - ***\*进程标识符\****：每个进程独一无二的 ID（父进程和子进程也靠这个关联）。
  - ***\*处理器状态信息\****：程序计数器（下一条执行哪句）、状态寄存器（如 x86 的 EFLAGS）和栈指针。
  - ***\*进程控制信息\****：进程现在的状态（运行还是阻塞？）、优先级、拥有哪些资源。
- ***\*考点避坑\****：PCB 极度重要，一旦被破坏（比如被系统 bug 篡改），操作系统就会失去对该进程的控制。

------

## 第二模块：进程的“生老病死”状态机 (画图题/大题重灾区)

这部分必考画图或状态转移判断！必须拿下！

***\*1. 初级形态：双态模型\****

最简单的理解，进程要么在***\*运行态 (Running)\****，要么在***\*非运行态 (Not Running)\****。决定谁上处理器的那个小程序叫 ***\*分派器/调度器 (Dispatcher)\****。

***\*2.\**** ***\*🌟\**** ***\*进阶形态：五态模型 (必背画图)\****

非运行态被细化了，变成了五个状态：

- ***\*新建 (New)\****：刚被创建，还没完全进内存。
- ***\*就绪 (Ready)\****：万事俱备，只欠 CPU（只要分派器选中我，我立马能跑）。
- ***\*运行 (Running)\****：正在 CPU 上狂奔。
- ***\*阻塞 (Blocked)\****：***\*重点！\**** 进程在等待某个事件（比如等用户输入、等 I/O 读取），这时候***\*即使把 CPU 给它，它也跑不了\****。
- ***\*退出 (Exit)\****：进程结束。
- **【考点辨析】**：从“运行”到“就绪”是因为***\*时间片用完了 (Timeout)\****；从“运行”到“阻塞”是因为***\*申请 I/O 或等待事件 (Event Wait)\****。千万别搞混！

***\*3. 终极形态：挂起状态 (Suspend)\****

- ***\*痛点\****：内存空间是有限的。如果内存里所有进程都在“阻塞”等 I/O，CPU 就会闲着没事干。
- ***\*绝招\****：操作系统会把阻塞的进程***\*踢出主存，放到磁盘上\****，腾出地方给新进程，这个动作叫 ***\*交换 (Swapping)\****。
- ***\*挂起的特征\****：被挂起的进程不能立即执行（因为在磁盘上），除非被显式唤醒。
- ***\*状态分裂\****：由此诞生了 ***\*就绪/挂起 (Ready/Suspend)\**** 和 ***\*阻塞/挂起 (Blocked/Suspend)\**** 两个新状态。

------

## 第三模块：操作系统的“幕后管理学” (选择/简答区)

***\*1. 操作系统的“四大账本”\****

为了管理整个系统，操作系统维护了四张表，它们互相交叉引用：

- ***\*内存表\****：管内存分配和虚拟内存映射。
- ***\*I/O 表\****：管设备通道分配和传输状态。
- ***\*文件表\****：管文件存在哪、当前状态和属性。
- ***\*进程表\****：管所有进程的 PCB。

***\*2. 用户态 vs 内核态\****

处理器执行时有两副面孔：

- ***\*用户态 (User Mode)\****：低特权，普通用户程序在这跑，防止你乱搞破坏系统。
- ***\*系统态/内核态 (Kernel Mode)\****：高特权，操作系统的核心代码在这跑，能执行特权指令。
- ***\*怎么切换？\**** 当发生***\*系统调用、中断 (如时钟/IO) 或 陷阱 (如程序报错)\**** 时，程序计数器会跳转，CPU 从用户态切入内核态。

## 第四模块：模式切换与进程切换的终极辨析 (核心易错点)

考试极喜欢问：***\*模式切换和进程切换是一回事吗？绝对不是！\****

- ***\*模式切换 (Mode Switch)\****：只是 CPU 的权限变了（比如从用户态切到内核态去处理一个中断），但***\*底层跑的还是同一个进程\****。只需保存几条寄存器信息，开销极小。
- ***\*进程切换 (Process Switch)\****：这可是“大换血”！CPU 要从进程 A 彻底切换到进程 B。
  - ***\*七步走（必背流程）\****：1. 保存当前处理器的上下文 -> 2. 更新当前进程的 PCB（把状态改为就绪/阻塞） -> 3. 把 PCB 移到相应队列 -> 4. ***\*挑选一个新进程\**** -> 5. 更新新进程的 PCB -> 6. 更新内存管理数据结构 -> 7. 恢复新进程的上下文。开销巨大！

结合 UNIX 系统，讲师给大家圈几个常考的实战概念：

***\*1. UNIX 状态机的特殊形态\****

- ***\*僵尸态 (Zombie)\****：进程其实已经死透了（不再存在），但它还***\*留下了一条记录等待父进程来收尸（收集状态信息）\****，这就叫僵尸进程。

***\*2. 神奇的\**** `***\*fork()\****` ***\*创建大法\****

# 🚀 第四章 线程、SMP与微内核

## 第一模块：包工头与打工人 —— 进程 vs 线程 (必考概念/辨析题)

在早期的操作系统中，进程既要占地盘（拥有资源），又要干苦力（被调度执行）。但这样太笨重了，于是现代操作系统把这两个身份拆开了：

- ***\*进程 (Process) —— “包工头”\****：它是***\*资源所有权\****的单位（占有虚拟地址空间、内存、文件等）。
- ***\*线程 (Thread) —— “打工人”\****：它是**调度/执行（分派）**的单位。一个进程里可以有多个线程，它们共享包工头的地盘（内存、资源），但各自干各自的活。

***\*🌟\**** ***\*核心考点：为什么我们需要引入线程？（背诵“四大快”）\****

1. ***\*创建快\****：创建一个新线程比新进程快得多（不用重新分配虚拟地址空间）。
2. ***\*销毁快\****：终止线程比终止进程快。
3. ***\*切换快\****：同一进程内的线程切换，不需要清空CPU缓存和TLB，开销极小。
4. ***\*通信快（免内核）\****：因为同进程的线程共享内存，它们互相传数据直接读写内存就行，***\*不需要调用内核\****。

------

## 第二模块：两大门派的对决 —— ULT vs KLT (🔥 全卷最高频大题)

线程具体是怎么被管理的？考试最爱考这两种线程架构的***\*优缺点对比\****：

### 用户级线程 (ULT - User-Level Threads)

- ***\*谁来管\****：完全由用户空间的***\*线程库\****（Threads Library）来管理，***\*内核根本不知道这些线程的存在\****（内核眼里只有一个进程）。
- ***\*优点（快、自由）\****：
  - 线程切换在用户态完成，***\*不需要切换到内核模式\****，速度极快。
  - 调度算法可以由应用程序自己定制。
  - 可以在任何操作系统上运行（只要有对应的线程库）。
- ***\*致命缺点（考试最爱考）\****：
  - ***\*“一人拉胯，全家连坐”\****：因为内核只认进程，如果其中一个线程执行系统调用被阻塞了，***\*整个进程里的所有线程都会被内核一起阻塞\****。
  - ***\*不能利用多核\****：纯ULT策略下，即使是多核CPU，内核一次也只能把这个进程分配给一个核心，里面的多个线程无法真正并行。
- ***\*补救措施\****：使用**套管（Jacketing）**技术，把阻塞系统调用转化为非阻塞调用。

### 内核级线程 (KLT - Kernel-Level Threads)

- ***\*谁来管\****：所有的线程管理都由***\*操作系统内核\****亲力亲为。
- ***\*优点（真正的并发）\****：
  - ***\*完美利用多核\****：内核可以把同一个进程的不同线程，同时分派到不同的处理器上并行执行。
  - ***\*独立阻塞\****：一个线程阻塞了，内核可以继续调度该进程的其他线程去执行。
- ***\*致命缺点\****：
  - ***\*慢（开销大）\****：同一进程内的线程切换，也必须切换到***\*内核模式（特权模式）\****，开销远大于ULT。

------

## 第三模块：多核时代的引擎 —— SMP 对称多处理 (选择/填空)

- ***\*它的前世今生 (Flynn分类法)\****：在计算机架构中，SMP 属于 ***\*MIMD（多指令多数据流）\**** 下的“共享内存（紧密耦合）”分支。
- ***\*SMP 的特征\****：内核可以在***\*任何处理器\****上执行，处理器会从池子里自己找活干。
- ***\*SMP 操作系统的设计难点 (考简答)\****：

由于多个CPU同时跑，内核代码必须是***\*可重入的\****，必须解决***\*调度分配、互斥同步（防止数据被同时改写）、存储器一致性以及容错性\****等复杂问题。

------

## 第四模块：操作系统的“极简风” —— 微内核 (Microkernel)

传统操作系统是个大杂烩，什么都塞在内核里。而微内核理念是**“断舍离”**。

- ***\*核心理念\****：内核只保留最基础的服务（如低级存储器管理、IPC消息传递），其他像文件系统、设备驱动、窗口系统，全部踢出内核，作为外围服务运行。
- ***\*优点 (4大特性)\****：
  1. ***\*统一界面\****：大家全靠***\*消息传递 (Message Passing)\**** 沟通。
  2. ***\*灵活性/扩展性\****：加功能、减功能都不需要动内核。
  3. ***\*可移植性\****：换了新硬件，只要改微内核那一点点代码就行。
  4. ***\*分布式系统支持\****：发消息给本地服务和远程服务，在微内核看来是一样的。
- ***\*致命缺点\****：***\*性能损失！\**** 因为组件之间疯狂发消息，开销比传统系统调用大得多。

## 第五模块：实战大PK —— 各大系统的独门绝技 (常识连线题)

这部分考得很细，重点记住这几个系统的“骚操作”：

1. ***\*Windows 的“亲和性”\****
   - ***\*软亲和性\****：调度器尽量把线程安排在它上次跑的那个CPU上（为了利用还没被冲刷掉的***\*L1/L2缓存数据\****）。
   - ***\*硬亲和性\****：强制规定某个线程只能在特定的某个CPU上跑。
2. ***\*Solaris 的“四层汉堡”与“神级中断”\****
   - 它为了融合 ULT 和 KLT，搞了个中间层叫 ***\*轻量级进程 (LWP)\****，负责把用户级线程映射到内核级线程上。
   - ***\*极其特殊的考点\****：Solaris ***\*把硬件中断也当作线程来处理\****！并且中断线程的优先级***\*高于所有其他类型的内核线程\****。
3. ***\*Linux 的“障眼法”\****
   - ***\*绝对考点\****：Linux 底层***\*根本不区分进程和线程\****！它统称它们为“任务 (Task)”，由 `task_struct` 结构管理。
   - ***\*怎么实现线程？\**** 它通过一个神仙系统调用 `***\*clone()\****`。它本质上是创建一个新进程，但可以通过标志位（如 `CLONE_VM`, `CLONE_FILES`）让新进程和老进程***\*共享同一个内存地址空间和文件表\****。共享了资源的进程，在表现上不就是线程了吗！

# 🚀 第五章：并发的互斥与同步

## 第一模块：认清敌人——并发为什么会出Bug？

在单核或多核CPU中，多个程序交错或同时运行，因为***\*执行的相对速度无法预测\****，就会产生并发问题。

- ***\*核心痛点：竞争条件 (Race Condition)\****

当多个进程读写同一个共享数据时，最终的结果完全取决于它们执行的先后顺序，这就叫竞争条件。比如两个线程同时对变量进行加减，最后的值可能完全不对。

- ***\*基础武器：原子操作 (Atomic Operation)\****

要解决并发问题，底层必须依赖“原子操作”：***\*它要么完全执行到底，要么根本不执行，绝对不会在中间被打断\****。

- ***\*生动案例：喂猫问题 (Feeding the Cat)\****

PPT里用了一个生动的例子：你和室友共同养了一只猫，猫每天只能喂一次。如果不加限制，你们俩可能会把猫撑死（重复喂）或者饿死（互相以为对方喂了）。为了解决这个问题，我们需要引入“锁”的概念。

## 第二模块：必考名词解释（考前默写区）

考试极其喜欢考这些概念的辨析，请死死记住这张表：

1. ***\*临界区 (Critical Section)\****：程序中访问共享资源的那段代码。***\*一次只能有一个进程进去\****。
2. ***\*互斥 (Mutual Exclusion)\****：当一个进程在临界区时，绝对不允许其他进程进入。
3. ***\*死锁 (Deadlock)\****：两个或多个进程互相死等对方释放资源，导致谁也无法继续。
4. ***\*活锁 (Livelock)\****：进程没有被阻塞，但一直在疯狂改变状态互相谦让，导致谁都干不了正事（比如Dekker算法的早期版本就容易出现）。
5. ***\*饥饿 (Starvation)\****：某个可运行的倒霉进程，被调度器无限期地忽略，永远得不到执行。

## 第三模块：从“石器时代”到“青铜时代”的解决方案

为了实现互斥，计算机科学家们尝试了软件和硬件的两套方案。这部分常考***\*优缺点辨析\****：

### 纯软件方案 (烧脑且易错)

- ***\*Peterson 算法\****：通过 `flag`（表示意愿）和 `turn`（表示谦让）完美实现了互斥，且比早期的 Dekker 算法更容易证明正确性。
- ***\*Lamport 面包店算法 (Bakery Algorithm)\****：专门解决N个进程的互斥。思想源于排队取号：票号最小的先进去；如果票号相同，进程 ID 小的优先（字典序比较）。

### 硬件级别的支持 (简单粗暴)

- ***\*禁用中断\****：进程进临界区前直接关闭中断。***\*致命缺点\****：把关中断特权交给用户很危险，且在多处理器（多核）上完全无效。
- ***\*专用机器指令 (\****`***\*testset\****` ***\*/\**** `***\*exchange\****`***\*)\****：硬件层面提供的一气呵成、绝对不会被打断的指令，可以在多处理器上使用。
  - ***\*🚨\**** ***\*终极考点（机器指令的缺点）\****：
    1. 会产生**“忙等待” (Busy Waiting)**，拿不到锁的进程会在门口疯狂死循环，白白浪费CPU时间。
    2. 可能导致某些进程永远抢不到锁，发生***\*饥饿\****。
    3. 如果有优先级机制，还可能引发死锁。

------

## 第四模块：黄金时代的“三大法宝”（期末大题轰炸区！！！）

因为硬件指令有“忙等待”的缺陷，操作系统为我们封装了三大高级同步原语。这是本章的绝对核心！

### 法宝一：信号量 (Semaphore) —— 考代码题最多！

- ***\*本质\****：一个包含非负整数值和等待队列的结构体。
- ***\*两大原子操作\****：
  - ***\*P 操作 (\****`***\*semWait\****`***\*)\****：测试。值减1，如果值 < 0，进程***\*把自己扔进队列睡眠（阻塞）\****。
  - ***\*V 操作 (\****`***\*semSignal\****`***\*)\****：增加。值加1，如果值 <= 0，从队列里***\*唤醒\****一个进程。
- ***\*强/弱信号量\****：如果等待队列采用***\*先进先出 (FIFO)\****，就是强信号量（能避免饥饿）；如果不规定顺序，就是弱信号量。
- ***\*🚨\**** ***\*必考大题模型：生产者/消费者问题\****
  - 要用三个信号量：互斥锁 `s=1`，商品数 `n=0`，空位数 `e=缓冲区大小`。
  - ***\*考试致命避坑点\****：***\*申请资源的 P 操作（\****`***\*semWait(n)\****` ***\*或\**** `***\*semWait(e)\****`***\*），必须永远写在申请互斥锁的 P 操作（\****`***\*semWait(s)\****`***\*）前面！\**** 否则如果进去后发现没资源，抱着互斥锁睡觉，必然导致全系统***\*死锁\****！而 V 操作的顺序则无所谓，只影响效率。

### 法宝二：管程 (Monitors) —— 傻瓜式的高级锁

- ***\*动机\****：信号量的 P/V 操作散落在代码各处，极其容易写错导致死锁。
- ***\*机制\****：管程把共享数据和方法封装在一起。天生自带互斥（任何时候只有一个进程能在管程里）。同步则依赖***\*条件变量\****。
- ***\*常考流派对比\****：
  - ***\*Hoare 管程\****：唤醒别人后，被唤醒者立刻执行。缺点是调度开销大，要求系统极其可靠。
  - ***\*Mesa 管程\****：使用 `cnotify` 和 `cbroadcast`。发出通知后，发信号的进程继续执行；被唤醒的进程进入就绪队列，***\*等真正轮到它时，必须重新检查一次条件\****（错误更少，模块化更好）。

### 法宝三：消息传递 (Message Passing) —— 分布式必备

- ***\*动机\****：管程搞不定多台计算机之间的同步，必须用消息传递。它同时满足***\*同步和通信\****两大需求。
- ***\*最常用组合\****：***\*无阻塞 Send（发完就去干别的） + 阻塞 Receive（死等消息到来）\****。
- ***\*寻址方式\****：可以通过信箱 (Mailbox) 进行间接寻址，实现更灵活的多对多通信。

# 🚀 操作系统期末冲刺绝密讲义：死锁与饥饿

## 第一模块：认清敌人——什么是死锁？(选择/判断高频区)

要解决问题，首先要认清问题。

***\*1. 死锁的本质\****

- ***\*绝对定义\****：死锁是一组竞争系统资源或相互通信的进程被***\*永久阻塞\****。每个进程都在等待另一个同组进程占用的某些资源，导致大家一起卡死。
- ***\*考试避坑\****：目前在系统设计中并没有绝对完美的有效解决方案。

***\*2. 资源的“两大门派”\****

考试极爱考资源的分类，记住以下两个特征：

- ***\*可重用资源\****：一次仅一个进程安全使用，且***\*不会因使用而耗尽\****（例如：处理器、主存、文件、数据库和信号量）。
- ***\*可消耗资源\****：可以被动态地***\*创建和销毁\****（例如：中断、信号、消息和I/O缓冲区中的信息）。如果进程间都在互相等待对方发送消息，就会引发可消耗资源死锁。

------

## 第二模块：召唤死锁的“四大条件”(必背大题)

这四个条件就像召唤神龙的四颗龙珠，***\*四个条件一起构成了死锁的充分必要条件\****：

1. ***\*互斥 (Mutual exclusion)\****：一次只有一个进程可以使用一个资源。
2. ***\*占有且等待 (Hold and wait)\****：当一个进程等待其他进程时，继续占有它已经分配到的资源。
3. ***\*不可抢占 (No preemption)\****：不能强行抢占进程已占有的资源。
4. ***\*循环等待 (Circular wait)\****：存在一个闭合的进程链，每个进程至少占有下一个进程所需的一个资源。前三个条件是发生死锁的必要条件，而第四个条件实际上是前三个条件的潜在后果。

------

## 第三模块：三大战役——如何处理死锁？(核心应用/简答题)

操作系统对付死锁有三套截然不同的武功心法，考试最爱考它们的对比！

### 战法一：死锁预防 (Deadlock Prevention) —— 严防死守的“保守派”

- ***\*核心思想\****：设计一个不可能出现死锁的系统，直接破坏四个必要条件之一。
- ***\*破解招式与致命缺点\****：
  - **破“互斥”**：不可行，因为有些资源天生必须互斥。
  - **破“占有并等待”**：要求进程一上来就***\*一次性请求所有所需资源\****。缺点是极度浪费资源，且系统往往不知道进程未来需要什么。
  - **破“不可抢占”**：如果进程请求被拒，必须释放已有资源。缺点是仅当状态可以轻松保存并稍后恢复时才实用（比如处理器），对其他资源不实用。
  - **破“循环等待”**：给资源定义***\*线性排序\****，只能按固定顺序申请。缺点是会不必要地拒绝资源，缺乏灵活性。

### 战法二：死锁避免 (Deadlock Avoidance) —— 走钢丝的“聪明人”

- ***\*核心思想\****：允许前三个条件存在，但在每次分配资源前，系统会***\*动态做出明智的选择\****以确保永远不会达到死锁点。它比“预防”允许更多的并发。
- ***\*🌟\**** ***\*终极考点：银行家算法 (Banker's Algorithm)\****
  - ***\*打个比方\****：把操作系统看作银行，进程看作借钱的客户，资源就是钱。银行保证在客户请求时，永远不会离开“安全状态”。
  - ***\*安全状态 vs 不安全状态\****：如果系统能找到至少一个资源分配序列不会导致死锁，就是***\*安全状态\****。***\*注意避坑：不安全状态不等于死锁状态！\**** 它只是预见到可能出现死锁，提前拒绝借钱。
  - ***\*算法前提（极其严苛）\****：必须事先知道每个进程需要的***\*最大资源量\****（Claim矩阵）、当前分配的资源量（Allocation矩阵）和系统当前的可用资源（Available向量）。并且占有资源时进程不能退出。

### 战法三：死锁检测与恢复 (Deadlock Detection) —— 亡羊补牢的“乐天派”

- ***\*核心思想\****：非常自由，不限制资源访问，只要有可能就会把资源分配给进程。系统定期执行死锁检测算法，检测是否发生了循环等待。
- ***\*恢复绝招（怎么收拾残局）\****：一旦检测到死锁，就终止所有死锁进程，或者回滚、连续抢占资源。
- ***\*杀进程的挑选标准（选择题常客）\****：

系统“杀”进程是有原则的，专挑软柿子捏，比如：目前为止***\*消耗处理器时间最少、产生输出最少、分配资源最少、或优先级最低\****的进程。

## 第四模块：经典副本——哲学家就餐问题 (概念理解与实战)

- ***\*场景\****：5个哲学家围着圆桌吃饭，每人左右各有一把叉子。由于缺乏生活技能，每个哲学家必须***\*拿到两把叉子才能吃饭\****。
- ***\*死锁怎么发生的\****：如果5个人同时拿起左手的叉子，大家都在等右手的叉子，就会死锁。
- ***\*完美破解方案\****：
  - ***\*信号量限制人数\****：在门外加一个控制机制（信号量），***\*只有4个哲学家可以一起用餐\****。这样至少有1个人能拿到两把叉子，彻底防止了死锁。

## 第五模块：四大操作系统的并发兵器库 (连线/填空秒杀)

这部分纯属记忆概念，考试喜欢考不同操作系统对应了什么底层机制。

- ***\*管道 (Pipes)\****：先进先出的循环缓冲区，用于生产者-消费者通信。
- ***\*消息 (Messages)\****：带有类型的文本，利用消息队列进行传递。
- ***\*共享内存 (Shared memory)\****：***\*最快的进程间通信方式！\**** 但它有个致命考点：***\*互斥约束不是共享内存机制的一部分\****，必须由进程自己提供锁。

***\*2. Linux 系统\****,,

- ***\*原子操作\****：包括整数操作和位图操作，执行时绝对不会被中断。
- ***\*自旋锁 (Spinlocks)\****：***\*Linux保护临界区最常用的技术\****。如果拿不到锁，线程不会睡觉，而是***\*一直尝试（自旋）\****。它非常适合预计等待时间极短的场景。
- ***\*屏障 (Barriers)\****：强行保证执行指令的顺序。

***\*3. Solaris 系统\****

- 支持互斥锁、信号量、多读者单写者锁、条件变量等原语。

***\*4. Windows 7 系统\****

- ***\*用户模式临界区\****：这个机制***\*只能被单个进程中的线程使用\****。如果是多处理器系统，代码会先试图获取自旋锁，一直拿不到才会阻塞线程交出处理器。

# 🚀 操作系统期末冲刺绝密讲义：第七章 内存管理

## 第一模块：包租婆的“五大基本原则”（内存管理需求）

操作系统管理内存，必须满足以下五个绝对的要求（***\*必背简答/填空题\****）：

1. ***\*重定位 (Relocation)\****：程序在内存中换入换出时，位置是不固定的。系统必须能够把程序的逻辑地址转换成真实的物理地址。
2. ***\*保护 (Protection)\****：每个进程只能在自己的地盘活动，绝对不能越权去读写别人的内存区域。
3. ***\*共享 (Sharing)\****：在保证保护的前提下，允许不同进程共享同一份代码（比如多个进程同时运行同一个程序，内存里只需要存一份）。
4. ***\*逻辑组织 (Logical organization)\****：内存是线性的一维空间，但我们写的程序是分模块的。系统要支持模块化的保护和共享（***\*分段\****技术最适合满足此需求）。
5. ***\*物理组织 (Physical organization)\****：内存（小而快）和外存（大而慢）之间的数据移动必须由系统自动完成，绝对不能让程序员自己去管。

------

## 第二模块：出租房屋的“演进史”（内存分区与碎片）

这是本章***\*最重要的考点之一\****，一定要分清“两种分区”和“两种碎片”的区别。

***\*1. 固定分区 (Fixed Partitioning)\****

- ***\*做法\****：系统提前把内存划分为大小相等或不等的固定块。程序只能装入比它小的块里。
- ***\*致命缺点——内部碎片 (Internal Fragmentation)\****：如果一个 2M 的程序放进了 8M 的分区，剩下的 6M 空间别人也用不了，这就是白白浪费在“分区内部”的垃圾空间。

***\*2. 动态分区 (Dynamic Partitioning)\****

- ***\*做法\****：根据程序需要，要多大就给多大，量体裁衣。
- ***\*致命缺点——外部碎片 (External Fragmentation)\****：随着进程不断进进出出，内存中会产生大量微小的、不连续的空隙。虽然总容量够，但拼不到一起去，这就是“分区外部”的垃圾空间。
- ***\*解决绝招——压缩 (Compaction)\****：操作系统不时地移动进程，把它们挤到一起，腾出一大块连续的空闲内存。但缺点是***\*极度耗时且浪费 CPU 时间\****。

***\*3. 折中方案：伙伴系统 (Buddy System)（大题考点）\****

- ***\*核心逻辑\****：内存块的大小永远是 ***\*2 的幂次方\****（比如128K, 256K）。如果要分配 70K 的空间，系统会把 256K 一分为二变成 128K，再把其中一个分配出去。等空间释放时，相邻的“伙伴”会像消消乐一样重新合并成大块。

------

## 第三模块：动态分区的“三大选房策略”（选择题必考）

当系统里有多个空闲内存块时，新来的进程该选哪一块？（对应我们之前做过的习题7.5）

1. ***\*首次适配 (First-fit)\****：从头开始扫描，碰到***\*第一个\****足够大的就塞进去。
   - **特点**：通常是最快、最好的，但会让内存前端产生很多小碎片。
2. ***\*下次适配 (Next-fit)\****：从上次放置的位置***\*继续往下扫\****，碰到够大的就塞。
   - **特点**：容易导致内存末尾的大块空闲空间很快被切碎，需要更频繁地进行压缩。
3. ***\*最佳适配 (Best-fit)\****：扫描全局，选一个***\*大小最接近\****请求的空闲块。
   - **🔥** **考试避坑**：名字叫“最佳”，但***\*总体表现最差！\**** 因为它每次都切得很抠门，留下的碎片小到任何程序都用不了，导致最严重的碎片化。

------

## 第四模块：地址转换的底层逻辑（计算题/填空题）

- ***\*逻辑地址\****：相对于程序起始点的地址，比如“程序开头往后偏移 1500 字节”。
- ***\*物理地址（绝对地址）\****：内存条上实实在在的物理位置。
- ***\*硬件支持（两把锁）\****：
  - ***\*基址寄存器 (Base Register)\****：存程序的起始物理地址。
  - ***\*界限寄存器 (Bounds Register)\****：存程序的长度上限（为了防止越界报错）。
  - **转换公式**：物理地址 = 基址寄存器的值 + 逻辑地址（相对地址）。

------

## 第五模块：终极解决方案——分页与分段（全章最核心对比）

为了彻底消灭固定分区的“内部碎片”和动态分区的“外部碎片”，现代操作系统引入了这两种技术：

***\*1. 简单分页 (Paging)\****

- ***\*原理\****：把物理内存切成大小相等的***\*页框 (Frame)\****，把进程切成同样大小的***\*页 (Page)\****。这样一页刚好塞进一个页框里，不需要连续存放。
- ***\*碎片情况\****：***\*没有外部碎片\****。只在进程的最后一页有***\*极少量的内部碎片\****。
- ***\*地址结构\****：逻辑地址被拆分为两部分 = ***\*页号 + 偏移量\****。操作系统通过查***\*页表 (Page Table)\**** 来找到页框号。

***\*2. 简单分段 (Segmentation)\****

- ***\*原理\****：按照程序的逻辑结构（比如主函数一段、数据一段、栈一段）切分成***\*长度不等\****的片段（有最大长度限制）。
- ***\*碎片情况\****：***\*没有内部碎片\****，但存在***\*外部碎片\****（类似于动态分区）。
- ***\*优势\****：非常符合程序员的逻辑思维，天然支持模块化编程和数据保护/共享。

# 🚀 操作系统期末冲刺绝密讲义：第八章 虚拟内存

## 第一模块：揭开魔法的面纱——虚拟内存的核心概念 (必考名词解释/填空)

***\*1. 什么是虚拟内存？\****

- ***\*核心魔法\****：计算机的物理内存（实存）可能只有 8GB，但你却能同时跑几十个加起来需要几十GB的程序。为什么？因为操作系统只把程序***\*当前正在使用的那几页\****（常驻集 Resident Set）放进主存，其余的都放在磁盘上。
- ***\*三大地址概念\****：
  - ***\*虚拟地址\****：程序里生成的地址（假地址）。
  - ***\*实地址\****：内存中真实的物理地址。
  - ***\*地址转换\****：所有虚拟地址在运行时都要被动态翻译成实地址。

***\*2. 两个极端现象（重点考点）\****

- ***\*👿\**** ***\*系统抖动 (Thrashing)\****：如果你把太多程序塞进内存，导致每个程序分到的内存太少，系统就会疯狂地在磁盘和内存之间来回搬运数据，反而不干正事了（CPU利用率暴跌）。
- ***\*👼\**** ***\*局部性原理 (Locality)\****：怎么避免抖动？靠这个原理！它指出程序在一段时间内，访问的代码和数据往往***\*聚集在一小块区域\****。所以操作系统可以大胆猜测未来需要哪些页，提前准备好，不用频繁搬运。

------

## 第二模块：底层的硬件护法——页表与 TLB (高频选择/简答题)

虚拟内存要起作用，必须靠强大的数据结构和硬件来快速完成“地址翻译”。

***\*1. 页表太大了怎么办？（两招破解）\****

每个进程都有自己的页表。如果地址空间极大，页表自己就会占满内存。

- ***\*多级页表\****：就像书的目录。用一个“根页表”指向“用户页表”，把庞大的单层目录变成多层，不需要的时候只把根目录放进内存。
- ***\*倒排页表 (Inverted Page Table)\****：不管你有多少进程，这个表的大小只和***\*物理内存的页框数\****有关。它用哈希函数来查找，极大地节省了空间（用于 PowerPC 等架构）。

***\*2.\**** ***\*🌟\**** ***\*终极考点：转换检测缓冲区 (TLB) —— 地址翻译的“作弊小抄”\****

- ***\*痛点\****：因为页表也放在内存里，所以每次访问数据，CPU 都要访问***\*两次内存\****（一次查页表，一次取数据），速度直接慢一半。
- ***\*解法 (TLB)\****：在 CPU 里搞一个硬件缓存（TLB），专门存放***\*最近查过的页表项\****。
- ***\*运行逻辑（必考流程）\****：CPU 拿到虚拟地址，先去 TLB 里找。如果***\*命中 (TLB Hit)\****，直接算出物理地址取数据；如果***\*未命中 (TLB Miss)\****，再去内存里查慢速的页表，查完顺便把新条目更新到 TLB 里。

------

## 第三模块：内存管理的“大管家”——四大策略 (大题/计算题重灾区)

这部分是期末考试的绝对核心！操作系统如何管理这方寸之地的内存？一共有四大策略：

### 读取策略 (Fetch) —— 什么时候把书拿上桌？

- ***\*请求分页\****：不见兔子不撒鹰，只有缺页报错了，才去磁盘把这页读进来（刚启动时缺页极多，后面靠局部性原理变少）。
- ***\*预先分页\****：一次性连带把周围的好几页全读进来（利用磁盘连续读取快的特性）。

### 🌟 置换策略 (Replacement) —— 书桌满了，扔哪本？（必考推演题）

当发生缺页，但内存已满时，必须踢掉一页。

- ***\*OPT (最佳策略)\****：预知未来，踢掉***\*将来最长时间内不会被用到的页\****。性能无敌，但无法实现，仅做对比标准。
- ***\*LRU (最近最少使用)\****：踢掉***\*最久没有被访问\****的那一页。性能最接近 OPT，但每次访问都要记录时间，开销太大。
- ***\*FIFO (先进先出)\****：最简单，谁先来就先踢谁。***\*缺点\****：经常用的核心代码可能也会被盲目踢掉。
- ***\*Clock (时钟策略)\****：给每个页加一个“使用位(u)”。指针像钟表一样转，遇到 u=1 就变成 u=0 并跳过，遇到 u=0 直接踢掉！这是***\*性能和开销的最佳平衡\****。
- **(升级版)** ***\*改进的时钟策略\****：不仅看“使用位(u)”，还看“修改位(m)”。优先踢掉 `(u=0, m=0)` 的页（既没用过也没改过），这样连写回磁盘的时间都省了！找页的顺序极其严格，考前必须手推一遍！

### 驻留集管理 (Resident Set) —— 给每个人分多大地盘？

- ***\*分配方式\****：***\*固定分配\****（一开始说好给几页就给几页） vs ***\*可变分配\****（根据程序运行情况动态调整）。
- ***\*置换范围\****：***\*局部置换\****（只能踢自己进程的页） vs ***\*全局置换\****（可以抢其他进程的空闲页）。
- **(高阶考点)** ***\*工作集 (Working Set, W(t, Δ))\****：利用过去 Δ 窗口内的访问记录来预测未来需要多少页。

### 清除与加载控制 (Load Control) —— 宏观调控

- ***\*页缓冲\****：被踢掉的页不直接消失，而是放入“空闲链表”或“修改链表”缓存一下。万一马上又要用，瞬间就能找回，极大弥补了 FIFO 算法的缺陷。
- ***\*多道程序设计级别\****：程序太少，CPU闲置；程序太多，系统抖动。需要挂起进程时，系统一般挑“优先级最低的”、“缺页的”或“占地儿最大的”进程开刀。

## 第四模块：大厂怎么做？—— Linux 与 Windows 的实战应用 (选择题/连线题)

这部分考得很细，直接背核心特征：

- ***\*Linux 系统\****：
  - 采用***\*三级页表结构\****：页目录、页中间目录、页表。
  - ***\*内核内存分配\****：因为内核经常需要分配极其微小的内存块，普通的按页分配太浪费，Linux 采用了***\*懒惰伙伴系统 (Lazy Buddy System)\****，推迟合并，极大提高了小块内存分配效率。
- ***\*Windows 系统\****：
  - 32位系统下，把 4GB 空间一分为二，***\*2GB给用户，2GB给操作系统\****。
  - 页的三种状态：***\*可用、保留、提交\****。
  - 驻留集策略：使用***\*可变分配、局部范围\****的策略（给每个进程配个工作集来动态调整）。

# 🚀 操作系统期末冲刺讲义：第九章 单处理器调度

## 第一模块：调度家族的“三兄弟” (必考选择/填空)

操作系统的调度不是一步到位的，它按时间长短分为三个阶段。你必须分清它们各自的任务：

1. ***\*长程调度 (Long-term scheduling) —— “守门员”\****
   - ***\*核心任务\****：决定把哪些作业放进系统变成进程，控制系统的***\*并发度\****（内存里到底装多少个进程）。
   - ***\*特点\****：如果放进来的进程太多，每个进程分到的CPU时间就少。
2. ***\*中程调度 (Medium-term scheduling) —— “搬运工”\****
   - ***\*核心任务\****：也就是我们常说的***\*交换 (Swapping)\****。决定把谁换入内存、把谁换出到磁盘挂起。
3. ***\*短程调度 (Short-term scheduling) —— “分配器 (Dispatcher)”\****
   - ***\*核心任务\****：决定***\*下一次由哪个就绪进程来执行\****。这是执行***\*最频繁\****的调度。
   - ***\*触发时机（考点）\****：当时钟中断、I/O中断、操作系统调用或信号发生时，短程调度程序就会跳出来干活。

------

## 第二模块：评判调度的“KPI”与“两把基本刷子” (简答/概念辨析)

在学习具体算法前，先搞懂规则：

### 调度的 KPI 评判指标

- ***\*面向用户\****：最核心的是 ***\*周转时间 (Turnaround time)\****（从提交到完成的总时间，包含等待和执行时间）和 ***\*响应时间 (Response time)\****（交互式系统中，从提交到第一次给出响应的时间）。
- ***\*面向系统\****：最核心的是 ***\*吞吐量 (Throughput)\****（单位时间完成的进程数）和 ***\*处理器利用率\****。

### 两把基本刷子：非抢占 vs 抢占 (Decision Mode)

这是所有调度算法的底层基因，考试选择题必考：

- ***\*非抢占 (Non-preemptive)\****：一旦进程开始跑，除非它自己运行完毕或请求I/O被阻塞，否则***\*绝对不会被打断\****。
- ***\*抢占 (Preemptive)\****：当前运行的进程可能随时被操作系统强行中断（比如时间片到了，或者来了个更高优先级的进程），把它扔回就绪队列。

## 第三模块：👑 六大核心调度算法 (本章终极大Boss：必考计算大题)

这一部分请竖起耳朵，期末必考***\*填表和画调度图\****。我们假定：$w$=已等待时间，$e$=已执行时间，$s$=需要的总服务时间。

### 先来先服务 (FCFS - First-Come-First-Served)

- ***\*原理\****：谁先排队谁先上（选择在队列中停留时间最长的，$max[w]$）。
- ***\*特点\****：***\*非抢占式\****。
- ***\*致命缺点\****：对短进程极度不友好。一个短进程排在一个长进程后面，会等得望眼欲穿。同时它***\*严重惩罚了I/O密集型进程\****。

### 轮转 (RR - Round Robin)

- ***\*原理\****：基于时钟中断的***\*抢占式\****算法。大家排好队，每个人只允许跑一个“时间片 (Quantum)”。时间到了就强行踢回队尾。
- ***\*避坑考点\****：***\*时间片的大小极其关键\****。时间片太短会导致频繁切换，开销极大；时间片太长就会退化成FCFS。一般建议时间片要***\*略大于一次典型交互的时间\****。
- ***\*进阶版：虚拟轮转法 (VRR)\****：为了解决传统RR对I/O密集型进程不公平的问题，VRR 引入了一个***\*辅助队列\****。从I/O阻塞中恢复的进程放进辅助队列，优先被调度，补偿它们之前没用完的时间片。

### 最短进程优先 (SPN - Shortest Process Next)

- ***\*原理\****：***\*非抢占式\****。谁需要的服务时间最短（$min[s]$），就让谁插队先跑。
- ***\*致命缺点\****：虽然极大改善了整体周转时间，但是长进程面临***\*饥饿 (Starvation)\**** 的风险，可能永远轮不到它。而且操作系统很难事先知道一个进程到底需要跑多久。

### 最短剩余时间 (SRT - Shortest Remaining Time)

- ***\*原理\****：***\*SPN 的抢占版\****。如果新来了一个进程，它需要的总时间，比当前正在跑的进程的“剩余时间 ($s-e$)”还要短，就立刻抢占CPU。
- ***\*特点\****：周转时间比SPN更优，但长进程依然会饥饿。

### 最高响应比优先 (HRRN - Highest Response Ratio Next) —— “养老尊贤”

- ***\*原理\****：***\*非抢占式\****。每次调度时，计算每个进程的响应比：***\*$Ratio = (w + s) / s$\****（等待时间+服务时间，除以服务时间），选最大的。
- ***\*精妙之处（简答题常客）\****：它完美结合了FCFS和SPN的优点。短作业的 $s$ 小，初始 Ratio 高，容易被选中；而长作业虽然 $s$ 大，但随着它等的时间 $w$ 越来越长，它的 Ratio 也会不断增加（***\*年龄老化机制\****），最终一定会被选中，***\*彻底解决了饥饿问题\****！

### 反馈调度 (Feedback) —— “动态降级”

- ***\*原理\****：***\*抢占式\****，多级队列机制。系统根本不需要提前知道进程需要跑多久。
- ***\*玩法\****：新来的进程进最高优先级队列（例如RQ0）。如果在它的时间片内没跑完，就被“惩罚”***\*降级\****，掉入下一个低优先级队列（RQ1...RQn）。越是长作业，最后掉得越深，在底层队列里使用轮转调度。

**(注：以上六大算法的对比总结表在讲义第7页，考试前一定要在脑子里默写一遍！)**

## 第四模块：其他现代调度机制 (填空/连线题)

1. ***\*公平共享调度 (Fair-Share Scheduling)\****
   - ***\*核心思想\****：不以“进程”为单位分配，而是以“用户”或“用户组”为单位进行份额分配。
   - ***\*公式逻辑\****：如果一个组占用的CPU超出了它的公平份额，这个组里所有进程的优先级就会被降低。
2. ***\*传统 UNIX 调度\****
   - ***\*机制\****：主要针对分时交互环境，采用多级反馈队列+轮转法（1秒抢占）。
   - ***\*优先级顺序（必背排序）\****：优先级从高到低依次为：***\*交换程序 > 块I/O设备控制 > 文件操作 > 字符I/O设备控制 > 用户进程\****。系统非常偏爱I/O操作，以求快速响应。

# 🚀 操作系统期末冲刺绝密讲义：第十章 多处理器与实时调度

## 第一模块：多处理器调度 —— “多核团队怎么干活？”

这一部分重点考察不同粒度的概念和线程的分配策略。

***\*1. 并行性的“五大粒度” (常考选择填空)\****

这就好比工作的拆分精细度：

- ***\*细粒度\****（<20条指令）：极短的代码级并行，极其复杂。
- ***\*中粒度\****（20-200条指令）：单应用程序内的多线程交互。
- ***\*粗粒度\****（200-2000条指令）：单核里多道程序的并发。
- ***\*极粗粒度\****（2000-1M条指令）：分布式网络节点运算。
- ***\*无约束\****：各个进程之间毫无瓜葛（比如分时系统里你打字、我听歌）。

***\*2. 处理器分配的两大阵营 (对比题)\****

把任务分给CPU，谁说了算？

- ***\*主从架构 (Master/Slave)\****：一个CPU当“包工头（主）”管调度，其他CPU当“打工人（从）”干活。***\*缺点\****是包工头一旦挂了，全盘崩溃，且容易成为性能瓶颈。
- ***\*对等架构 (Peer)\****：没有阶级，所有CPU自己去任务池里抢活干。***\*缺点\****是操作系统设计极其复杂，得防止两个CPU抢到同一个任务。

***\*3.\**** ***\*🌟\**** ***\*核心考点：多处理器线程调度的四大绝招\****

这是大题/简答题的高频区，必须记住它们的名字和特点：

- ***\*负载分配 (Load Sharing)\****：最简单常用。所有CPU共用一个全局任务队列（像银行排号排一队）。***\*缺点\****是全局队列容易成瓶颈，且缓存命中率低（因为线程可能在不同CPU上横跳）。
- ***\*组调度 (Gang Scheduling)\****：***\*“打团战”模式\****。同一个程序的所有线程被***\*同时\****调度到不同的处理器上一起运行。这样可以极大减少线程间互相等待的阻塞时间。
- ***\*专用处理器分配 (Dedicated Processor)\****：***\*“VIP包房”模式\****。一个程序运行期间，分给它的CPU就一直死死绑定给它，哪怕线程在等I/O，CPU也宁可闲着。适用于有成百上千个CPU的土豪系统，省去了进程切换的开销。
- ***\*动态调度 (Dynamic Scheduling)\****：OS负责分CPU，应用程序自己根据情况动态增减线程数量，极其灵活。

## 第二模块：实时调度 —— “十万火急的任务怎么插队？”

实时系统（如航天器、工厂控制）不仅要求结果算得对，更要求**“按时交卷”**。晚一秒可能就是机毁人亡！

***\*1. 两大任务类型 (基础概念)\****

- ***\*硬实时 (Hard real-time)\****：***\*绝对不能迟到\****，迟到系统就彻底崩溃或引发灾难（比如汽车安全气囊）。
- ***\*软实时 (Soft real-time)\****：有最后期限，最好能按时做完，***\*迟到了也没关系\****，只是体验差一点（比如视频直播卡顿）。

***\*2. 实时OS的“五大修养” (简答题背诵)\****

什么样的系统才配叫实时操作系统？

- ***\*可确定性\****：响应中断的时间是固定可预期的。
- ***\*可响应性\****：能光速处理中断并开始执行服务代码（从高优先级中断到开始服务通常只要几十到几百毫秒）。
- ***\*用户控制\****：允许用户自己定“谁是硬实时，谁是软实时”，权限给够。
- ***\*可靠性\****：死机就是灾难，必须稳如老狗。
- ***\*故障弱化操作\****：如果实在干不完了，系统必须优先保全最重要的任务，丢车保帅。

***\*3. 实时调度的四种流派 (连线题)\****

- ***\*静态表调度\****：运行前就安排好死板的时间表（适合周期性任务）。
- ***\*静态优先级抢占\****：事先排好VIP等级，高级来了直接抢占低级（如速率单调算法）。
- ***\*基于动态规划\****：任务来了临时算一下能不能排得开。
- ***\*动态尽力而为\****：不废话直接干，干不完就直接杀掉进程（最容易实现）。

***\*4.\**** ***\*🌟\**** ***\*速率单调算法 (RMS)\****

这是一个具体的静态优先级算法：***\*任务的周期越短，执行频率越高，优先级就越高\****。

## 第三模块：🔥 终极Boss —— 优先级反转 (期末必考大题)

如果这张卷子只考一道大题，大概率就是它！

***\*1. 什么是优先级反转 (Priority Inversion)？\****

- ***\*故事背景（火星探路者号故障）\****：高优先级任务 T1 和低优先级任务 T3 要共享一个临界资源（比如一块内存）。
- ***\*灾难发生\****：T3 先拿到了资源锁。T1 来了，发现资源被锁，只能等待。结果，一个中等优先级任务 T2 跑出来***\*抢占了 T3\****。这就导致：中等优先级 T2 在爽，低优先级 T3 被挂起锁着资源，高优先级 T1 被迫干瞪眼死等！这就叫***\*无界优先级反转\****。

***\*2. 两大破解之法 (核心解法)\****

- ***\*优先级继承 (Priority Inheritance)\****：T1 发现自己被 T3 堵住了，就***\*临时把自己的高优先级借给（继承）T3\****，让 T3 赶紧跑完释放资源。T3 跑完后，优先级打回原形，T1 拿到资源继续跑。
- ***\*优先级置顶 (Priority Ceiling)\****：提前规定好，这块共享资源的优先级，比所有可能会用它的任务的优先级***\*还要高一级\****。谁拿到这个资源，优先级瞬间飙升到顶点，直到用完才恢复。

# 🚀 操作系统期末冲刺讲义：第11章 I/O管理和磁盘调度

## 第一模块：I/O控制的三大绝招与“神级外包” DMA (必考选择/简答)

外设种类繁多，速度差异极大（键盘极慢，网络极快）。为了让飞快的CPU不被慢吞吞的外设拖后腿，I/O控制技术经历了三次进化：

1. ***\*程序控制I/O\****：CPU亲自盯着设备，一直死等（忙等待），极度浪费CPU时间。
2. ***\*中断驱动I/O\****：CPU下达命令后去干别的，设备干完活发个“中断（短信）”通知CPU。
3. ***\*🌟\**** ***\*终极考点：DMA（直接存储器访问）\****
   - ***\*打个比方\****：CPU是大老板，DMA就是“专门负责搬砖的外包队长”。
   - ***\*核心逻辑\****：CPU只需要在***\*开始和结束时\****介入。CPU把要读写的地址和字数告诉DMA，DMA就直接控制内存和设备之间的数据传输，完全不需要CPU插手。传完之后，DMA再给CPU发个中断报告“活干完了”。
   - ***\*考试避坑\****：DMA不是不需要CPU，而是把CPU的干预降到了最低！

## 第二模块：缓冲技术 (Buffering) —— 操作系统的“蓄水池”

为什么需要缓冲？因为I/O速度跟不上处理器，不能让进程干等。

考试常考缓冲分类及其特性：

- ***\*设备分类\****：
  - ***\*面向块（Block）\****：数据一块一块传（如磁盘、U盘）。
  - ***\*面向流（Stream）\****：数据一字节一字节传，没结构（如打印机、终端键盘）。
- ***\*缓冲方案对比\****：
  - ***\*单缓冲 (Single Buffering)\****：系统分出一个缓冲区，设备一边往里写，进程一边从里读。
  - ***\*双缓冲 (Double Buffering)\****：又叫***\*缓冲交换\****。设两个蓄水池，设备装满A池时，进程可以去喝B池的水，完美交替。
  - ***\*循环缓冲 (Circular Buffering)\****：两个以上缓冲区首尾相连，专门应对I/O需求爆发的场景。

## 第三模块：磁盘调度算法 (💥 绝对重点！必考大题推演)

这就是期末考试绝对会出的一道10-15分大题！系统有一堆读写磁道的请求，先去哪个后去哪个，决定了磁盘的生死存亡。

***\*1. 磁盘时间的“三剑客”\****

- ***\*寻道时间 (Seek Time)\****：磁头移动到正确***\*磁道\****的时间。（这个最耗时，是所有算法优化的唯一目标！）
- ***\*旋转延迟 (Rotational Delay)\****：等磁盘转到正确***\*扇区\****的时间。
- ***\*传输时间 (Transfer Time)\****：真正读写数据的时间。

***\*2. 必背五大调度算法\****

假设磁头现在在100道，请求队列是：55, 58, 39...

- ***\*FIFO (先进先出)\****：谁先来先服务谁。最公平，但磁头可能满天飞，性能极差。
- ***\*SSTF (最短服务时间优先)\****：***\*永远找离当前磁头最近的\****磁道。
  - **缺点**：会导致“饥饿”，边缘的磁道永远等不到磁头。
- ***\*🌟\**** ***\*SCAN (扫描算法 / 电梯算法)\****：***\*必考！\****
  - **逻辑**：就像电梯，先一直往一个方向走（比如朝大数走），把路上的请求全扫完，***\*碰壁后（到底了）再掉头\****往回扫。
  - **优点**：比SSTF公平，防止了饥饿。
- ***\*C-SCAN (循环扫描)\****：电梯只在一楼接人上楼。扫到顶后，***\*直接飞速空车返回底部\****，再从底向上扫，让两端的等待时间更均匀。
- ***\*N-step-SCAN & FSCAN\****：为了防止磁头在某个磁道“粘住”（不断有新请求落在当前磁道），把队列分成子队列，扫描时新来的请求只能进下一个队列排队。

## 第四模块：RAID 磁盘阵列 (选择/填空重灾区)

RAID（独立磁盘冗余阵列）的核心思想是：***\*“鸡蛋不要放在一个篮子里”\****。用多块便宜的磁盘组合起来，速度快又防崩溃。

你必须记住以下几个级别的核心特征：

- ***\*RAID 0 (条带化, 无冗余)\****：不考虑安全，纯追求速度。数据拆分存在不同盘上，一旦坏一块盘，数据全毁。
- ***\*RAID 1 (镜像)\****：最土豪最安全。数据写两份（复制粘贴），坏了一块还有备份。成本最高（100%冗余）。
- ***\*RAID 2 / 3 / 4\****：通过不同方式（如汉明码、奇偶校验盘）做恢复。缺点是如果只用一个盘做校验，那个校验盘会成为性能瓶颈。
- ***\*🌟\**** ***\*RAID 5 (块级分布式奇偶校验)\****：***\*企业最常用！必考！\**** 它把校验位***\*均匀分散\****在所有盘上，避免了RAID 4的校验盘瓶颈。坏任意一块盘都能通过其他盘算出来恢复。
- ***\*RAID 6 (双重冗余)\****：有两个不同的校验块，牛逼在***\*允许同时坏两块盘\****！代价是写入速度慢（有写损失）。

------

## 第五模块：磁盘缓存与置换策略 (概念辨析)

为了少读磁盘，我们在主存里划一块区域当“磁盘高速缓存(Disk Cache)”。缓存满了怎么踢人？

- ***\*LRU (最近最少使用)\****：把最久没被访问的块踢出去，用堆栈实现。
- ***\*LFU (最不常使用)\****：谁被访问的次数最少踢谁。缺点是容易被短时间内的集中访问“欺骗”。
- ***\*基于频率的置换 (Frequency-Based Replacement)\****：把堆栈分成“新、中、老”三区。在“新区”里的访问不增加计数器，只有跑到“老区”且计数最小的才会被踢。完美结合了LRU和LFU的优点。

## 第六模块：Linux 和 Windows 的 I/O 绝活 (常识连线题)

- ***\*Linux I/O 调度器\****：
  - ***\*Linus电梯\****：基本排序合并请求。
  - ***\*最后期限调度 (Deadline)\****：加了3个队列，加上了过期时间，防止请求饿死。
  - ***\*预期调度 (Anticipatory)\****：***\*很聪明的一招！\**** 读完一个扇区后，磁头***\*故意停下来等几毫秒\****，赌下一个请求就在旁边（因为程序的局部性），极大提高了性能。
- ***\*Windows I/O\****：
  - 提供***\*异步I/O\****（发起请求后不阻塞继续运行）和***\*同步I/O\****（死等）两种模式。
  - 提供多达5种完成信号技术（如信号通知、异步过程调用APC、I/O完成端口等）。

# 🚀 操作系统期末冲刺绝密讲义：第十二章 文件管理

## 第一模块：基本概念与文件的“五大阵型” (名词解释/选择题区)

***\*1. 数据的“套娃”层级\****

在文件系统中，数据是按照从小到大的层级组织的，考试常考这四个名词的递进关系：

- ***\*域 (Field)\****：数据的基本元素，包含单个值。
- ***\*记录 (Record)\****：一组相关域的集合，被视为一个单元。
- ***\*文件 (File)\****：相似记录的集合，作为单一实体对待，通过名称引用。
- ***\*数据库 (Database)\****：相关数据的收集，数据元素之间的关系明确。

***\*2.\**** ***\*🌟\**** ***\*必考核心：五种常见的文件组织形式\****

考试极爱考它们各自的特点和应用场景（连线题/选择题必背）：

- ***\*堆 (Pile)\****：最简单的形式，数据按到达的顺序收集，没有任何结构，只能通过***\*穷举搜索\****来找数据。
- ***\*顺序文件 (Sequential)\****：最常见，记录按“关键字”顺序存储，通常用于***\*批量应用\****。
- ***\*索引顺序文件 (Indexed Sequential)\****：在顺序文件基础上加了***\*索引和溢出文件\****，大大减少了查找时间。
- ***\*索引文件 (Indexed)\****：只能通过索引访问，包含详尽索引或部分索引；极其适合***\*信息及时性至关重要的应用\****（如机票预订、库存控制）。
- ***\*直接或散列文件 (Hashed)\****：利用对键值的散列，直接访问已知地址的块；适用于需要***\*非常快速的访问\****且一次只访问一条记录的场景（如目录、定价表）。

**注：现代数据库的索引经常使用** ***\**B树 (B-Tree)\**\*****，它的特点是平衡、宽且浅，搜索很快就能结束**,。

## 第二模块：记录怎么装箱？—— 记录组块 (简答题/对比题)

文件是由“记录”组成的，而磁盘 I/O 的读写单位是“块”。把记录塞进块里，有三种“装箱”策略：

1. ***\*定长组块 (Fixed Blocking)\****：使用固定长度的记录，完整的记录全塞在一个块里。
   - **致命考点**：块的末尾会有装不满的空间，产生***\*内部碎片\****。
2. ***\*变长跨越式组块 (Variable Blocking: Spanned)\****：记录紧缩在块中，不留空隙。如果一个记录太长，会***\*跨越两个块\****，中间用指针连起来。
   - **优点**：存储利用率极高，不浪费空间。**缺点**：实现起来非常困难。
3. ***\*变长非跨越式组块 (Variable Blocking: Unspanned)\****：记录变长，但绝不跨块。
   - **缺点**：会导致大量浪费的空间（内部碎片），并限制了记录的最大长度。

## 第三模块：地盘怎么分？—— 文件分配方法 (🌟 全章第一大重点)

文件要在磁盘上安家，操作系统怎么给它分地盘？这三种方法的优缺点对比是期末必考大题！

***\*1. 连续分配 (Contiguous Allocation)\****

- ***\*做法\****：创建文件时，给它分配一组***\*连续的块\****。文件分配表（FAT）里只需记下***\*起始块和长度\****。
- ***\*考点\****：对单个顺序文件来说性能最好，但会产生***\*外部碎片\****，必须靠“紧缩”来解决。

***\*2. 链式分配 (Chained Allocation)\****

- ***\*做法\****：基于单个块分配，每个块里都藏着一个***\*指向下一个块的指针\****。FAT 表同样只有一条记录。
- ***\*考点\****：完美解决了外部碎片问题，但***\*局部性原理不再适用\****（磁头可能满磁盘乱跑），需要周期性合并。最适合顺序处理。

***\*3. 索引分配 (Indexed Allocation)\****

- ***\*做法\****：FAT 表为每个文件建一个***\*单独的一级索引表\****。索引表里记录了文件占用的所有块的块号,。
- ***\*考点\****：它结合了前两者的优点，是极其重要的方法，Unix 系统就是用这种思路。

------

## 第四模块：剩下的地盘怎么管？—— 空闲空间管理 (选择/填空)

分配完文件后，磁盘上剩下的空闲块怎么管？有四种方法：

1. ***\*位表 (Bit Tables)\****：用一串 0 和 1 表示，每个位代表一个块（0表示空闲，1表示占用）。它非常小，容易找空闲块。
2. ***\*链接空闲区 (Chained Free Portions)\****：把所有空闲块用指针串起来。不需要额外的表，但会产生碎片，且每次分配都要读块来找指针，开销大,。
3. ***\*索引 (Indexing)\****：把空闲空间直接当成一个文件，用索引表来管理。
4. ***\*空闲块列表 (Free Block List)\****：把所有空闲块的序号存下来，放在主存里当作“栈”或者“FIFO队列”来管理。

## 第五模块：实战大厂怎么做？—— UNIX 与 Windows NTFS (大题扩展区)

这一部分主要考一些名词对应的操作系统，比如问你 `inode` 是谁家的，`VFS` 是什么。

***\*1. UNIX 的独门绝技\****

- ***\*索引节点 (Inode)\****：这是 UNIX 管理文件的核心！一个活跃的 Inode 控制着一个文件，里面包含了文件的所有元数据。
- ***\*分配方案\****：UNIX 在 Inode 中使用了极其精妙的设计：包含多个***\*直接指针\****，以及***\*一级、二级、三级间接指针\****。这使得它可以支持超过 16GB 的巨大文件。
- ***\*Linux VFS (虚拟文件系统)\****：它为用户提供了一个统一的接口，不管底层是什么文件系统，它都抽象成四个核心对象：***\*超级块对象\****、***\*索引节点对象\****、***\*目录项对象\****、***\*文件对象\****。

***\*2. Windows NTFS 的特点\****

- NTFS 设计用来满足高端需求，考点在于它的特性：***\*可恢复性、安全性（安全描述符）、大磁盘支持、日志、压缩和加密\****。
- ***\*存储单位\****：在 NTFS 中，磁盘的层级是 ***\*扇区 (Sector)\**** $\rightarrow$ ***\*簇 (Cluster，连续扇区)\**** $\rightarrow$ ***\*卷 (Volume，逻辑分区)\****。

# 🚀 操作系统期末冲刺绝密讲义：第十三章 分布式与集群

## 第一模块：客户-服务器（C/S）架构的“四种形态”与“三层境界” (选择/判断高频区)

***\*1. 谁是客户？谁是服务器？\****

- ***\*客户 (Client)\****：通常是你的个人PC，负责给你提供友好的图形界面（GUI、窗口、鼠标）。
- ***\*服务器 (Server)\****：背后的大佬，通常是高性能计算机，运行着关系数据库（DBMS），允许多个客户共享访问。
- ***\*网络\****：连接两者的第三个基本要素。

***\*2.\**** ***\*🌟\**** ***\*必考点：C/S 架构的四大门派（应用级任务怎么分配？）\****

考试极爱考这四种架构的特征对比：

- ***\*基于主机的处理 (Host-based)\****：***\*不是真正的C/S\****。主机包揽一切，客户机只是个“哑终端”（只负责显示）。
- ***\*基于服务器的处理 (Server-based)\****：服务器包干几乎所有处理，客户端只提供图形界面。好处是系统好维护。
- ***\*基于客户的处理 (Client-based)\****：***\*目前最普遍的模式！\**** 应用处理都在客户端完成，服务器只负责最核心的数据库逻辑。好处是用户可以高度定制本地应用。
- ***\*合作处理 (Cooperative)\****：两边“胖瘦搭配”，以最优化的方式协同处理。好处是生产和网络效率最高，***\*缺点是设置和维护极其复杂\****。

***\*3. 三层 C/S 架构 (Three-tier Architecture)\****

为了进一步解耦，现代系统变成了三层：

- ***\*用户机器\****（瘦客户端） $\rightarrow$ ***\*中间层服务器\****（应用服务器/网关，负责转换协议和融合结果） $\rightarrow$ ***\*后端服务器\****（数据服务器）。

------

## 第二模块：沟通的桥梁——中间件与缓存 (概念填空区)

***\*1. 中间件 (Middleware) —— 系统的“通用翻译官”\****

- ***\*痛点\****：客户和服务器的操作系统、硬件五花八门，程序员怎么写代码？
- ***\*大招\****：引入中间件！它提供一组标准的 API（应用程序编程接口），让程序员可以***\*忽略底层的跨平台差异\****，无论数据在哪都能用相同的方法访问。

***\*2. 文件高速缓存一致性 (File Cache Consistency)\****

- ***\*痛点\****：网络传输太慢了，每次都去服务器读文件会卡死。
- ***\*绝招\****：利用***\*局部性原理\****，把最近访问的文件缓存在本地。
- ***\*考点避坑\****：怎么保证本地缓存和服务器一致？最简单粗暴的方法是***\*文件锁技术\****，防止多人同时访问，但***\*代价是牺牲了性能和灵活性\****。

------

## 第三模块：进程如何跨机器聊天？(核心对比大题)

机器不在同一台物理机上，进程怎么通信？这是本章的绝对核心！

***\*1. 分布式消息传递 (Message Passing)\****

- ***\*可靠性 vs 不可靠性\****：可靠机制保证传输正确（有确认回复），不可靠机制只管发不管到（降低了复杂性和开销）。
- ***\*🌟\**** ***\*阻塞 vs 无阻塞（必背对比）\****：
  - ***\*阻塞 (Blocking / 同步)\****：发消息后死等，直到确认对方收到才继续运行。
  - ***\*无阻塞 (Nonblocking / 异步)\****：发完就去干别的事，***\*极其高效灵活\****，但***\*致命缺点是极难测试和调试\****（由于时间顺序不确定，会产生很多奇怪的bug）。

***\*2.\**** ***\*🌟\**** ***\*远程过程调用 (RPC, Remote Procedure Call)\****

- ***\*概念\****：就像调用本地函数一样，去调用另一台机器上的函数。它能自动生成通信代码。
- ***\*绑定方式对比\****：
  - ***\*非永久绑定\****：打完一次电话就挂断（连接/断开频繁，开销大，不适合频繁调用）。
  - ***\*永久绑定\****：电话一直通着，适合多次重复调用。
- ***\*考试大坑（参数传递的困难）\****：RPC传“值”很容易，但***\*传“引用（指针）”极其困难\****！因为两台机器的内存地址完全不一样，必须要实现系统范围内的全局指针，代价极高。而且两边编程语言不同也会导致数据格式不一致。

***\*3. 面向对象机制\****

- 引入***\*对象请求中介 (ORB)\**** 作为代理，客户和服务器之间互传对象消息。经典例子：微软的 COM 和 CORBA。

------

## 第四模块：终极武器——集群 (Clusters) (简答/优势列举区)

如果一台服务器扛不住，那就上一群服务器！

***\*1. 什么是集群？\****

一组互连的***\*完整计算机\****（注意：必须是离开集群也能独立运行的机器），像一台计算机一样协同工作。每一台叫一个“节点”。

***\*2.\**** ***\*🌟\**** ***\*集群的四大绝对优势（必背简答题）\****

1. ***\*绝对可伸缩性\****：几十上百台机器绑一起，性能远超最大的单体计算机。
2. ***\*增加可伸缩性（增量扩展）\****：随时往里添新机器，工作量极小。
3. ***\*高可用性\****：一台机器宕机了，其他机器顶上，系统不死。
4. ***\*高性价比\****：用市面上便宜的普通PC拼装，就能打败昂贵的大型机。

***\*3. 集群 vs 对称多处理器 (SMP) （选择/辨析题）\****

- ***\*SMP 的优点\****：容易管理配置，占地小，耗电少，稳定。
- ***\*集群的优点\****：在绝对规模和扩展性上，集群***\*完爆\**** SMP。

***\*4. 操作系统要解决的三个核心问题\****

- ***\*故障管理\****：故障补救 (Failover，把任务切给好机器) 和 故障恢复 (Failback，机器修好后切回来)。注意：自动 Failback 可能会导致任务来回“反弹”。
- ***\*负载平衡\****：新机器加入时自动分配任务。
- ***\*并行计算\****：依赖并行编译器、并行应用和参数计算（同算法不同参数）。

## 第五模块：实战演练——Windows、Sun 与 Linux 集群 (连线/常识题)

***\*1. Windows 故障转移集群\****

- 这是一种***\*不共享\****集群（资源在某时刻只归单个节点所有）。
- 核心概念：***\*集群服务\****（管家）、***\*资源\****（被管理的对象）、***\*组\****（按单位管理的资源集，用于故障补救和负载平衡）。

***\*2. Sun 集群\****

- 包含统一的对象框架、全局分布式文件系统等。

***\*3. Beowulf 和 Linux 集群（高频背景考点）\****

- ***\*身世\****：1994年NASA发起，目前最重要的集群技术。
- ***\*特征（专考选择题）\****：用***\*市面上常见的部件（COTS）\****，***\*专用的私有网络\****，***\*免费的开源软件库\****，绝不搞花里胡哨的定制硬件，极易复制。代表软件有 BPROC（分布式进程空间）。

# 🚀 操作系统期末冲刺绝密讲义：第14章 分布式进程管理

## 第一模块：跨电脑“搬家”—— 进程迁移 (Process Migration)

在分布式系统中，进程有时候跑着跑着，需要换一台电脑继续跑，这就是进程迁移。

***\*1. 为什么要搬家？（必背四大动机）\****

- ***\*负载共享\****：这台机器太卡了，去另一台闲置的机器跑。
- ***\*通信性能\****：数据在哪，进程就去哪（为了少传数据）。
- ***\*可用性\****：这台机器马上要关机维护了，赶紧跑路。
- ***\*特殊功能\****：目标机器上有特殊的硬件（比如高端GPU）或软件。

***\*2. 怎么搬家？（“四大搬家策略”选择题常客）\****

必须在原机器销毁，在目标机器创建。内存地址空间怎么搬？

- ***\*Eager (all) 全搬\****：简单粗暴，把所有地址空间一次性全传过去。（缺点：太慢，很多空间根本用不上）
- ***\*Eager (dirty) 脏搬\****：只传在主存里被**修改过（dirty）**的部分，剩下的按需传输。
- ***\*Copy-on-reference 按需搬\****：先过去，用到哪页再传哪页。（优点：初始启动最快）。
- ***\*Flushing 刷新\****：把脏页刷回磁盘，清空主存资源。
- **(扩展：*****\**预先复制 Precopy\**\*****：一边偷偷传数据，一边让进程继续跑，减少进程被冻结的时间)**

***\*3. 移出 (Eviction)\****

如果一个进程迁移到了一台闲置工作站，但工作站的主人突然回来了（开始操作电脑），为了保证主人的响应时间，必须把这个外来进程**“踢出去”（移出）**。

------

## 第二模块：拍合影的难题—— 分布式全局状态 (Global State)

在分布式系统里找Bug或者检测死锁，需要给整个系统拍个“快照”。但因为大家***\*没有统一的手表（公共时钟）\****，极容易出现时空错乱。

***\*1. 核心考点：一致性状态 vs 不一致状态\****

- ***\*一致状态 (Consistent)\****：发信和收信的历史是吻合的。
- ***\*不一致状态 (Inconsistent)（必考辨析）\****：***\*接收方已经记录了收到消息，但发送方居然还没记录发送这个消息！\****（就像你收到了快递，但淘宝上显示商家还没发货，典型的时空悖论）。

***\*2. 分布式快照算法\****

- ***\*秘密武器：标识 (Marker)\****。
- 进程通过向所有流出通道广播一个“标识”来启动算法。保证记录下大家的状态和通道里的消息。

------

## 第三模块：抢共享资源的终极PK—— 分布式互斥 (Mutual Exclusion)

既然没有共享内存，大家怎么抢同一把锁？***\*全靠互相发消息商量！\****

***\*1. 如何给事件排先后？（基于时间戳的算法）\****

大家既然没有公共时钟，就自己维护一个计数器。

- 发消息时带上自己的计数器：`(消息内容, 时间戳, 进程ID)`。
- 收到消息时，更新自己的时钟：`我的新时钟 = 1 + max(收到时间戳, 我的旧时钟)`。
- ***\*谁排前面？\**** 时间戳小的优先；如果时间戳一样，进程ID小的优先。

***\*2.\**** ***\*🌟\**** ***\*三大分布式互斥算法对比（全卷最高频考点！）\****

这是大题和选择题的重灾区，记住它们的优缺点和消息数量！

- ***\*① Lamport 算法（拉姆波特算法）\****
  - **原理**：大家都有一个请求队列。想进临界区，必须给所有人发 `Request`，等所有人回复 `Reply`。用完后发 `Release`。
  - **开销**：进一次临界区需要 ***\*3(N-1)\**** 条消息。（N-1个请求，N-1个回复，N-1个释放）。
- ***\*② Ricart & Agrawala 算法（优化版）\****
  - **原理**：去掉了释放消息。如果你收到了别人的请求，但你正在用，你就***\*装死不回复\****（推迟回复），等你用完了再发 `Reply`。
  - **开销**：优化到了 ***\*2(N-1)\**** 条消息。（N-1个请求，N-1个回复）。
- ***\*③ Token-Passing (令牌传递算法)\****
  - **原理**：系统里只有一个“令牌”，谁拿到令牌谁就能进临界区。用完按顺序传给下一个需要的人。
  - **开销**：如果你手里没令牌，需要 ***\*N\**** 条消息去请求；如果你手里刚好有令牌，***\*0条消息\****就能进！（性能波动大）。

------

## 第四模块：跨主机的“死结”—— 分布式死锁 (Distributed Deadlock)

大家互相发消息等资源，极其容易等成死胡同。

***\*1. 死锁预防的两个神技\****

遇到冲突时，系统通过比较两个进程的“时间戳（年龄）”来决定谁生谁死：

- ***\*Wait-die (等或死)\****：老进程可以等新进程；新进程如果遇到老进程占着资源，新进程直接自杀。
- ***\*Wound-wait (伤或等)\****：老进程可以直接“砍死（抢占）”新进程的资源；新进程遇到老进程，只能乖乖等。

***\*2. 消息通信中的死锁（Buffer满了）\****

不仅仅是等资源会死锁，***\*发消息也会死锁\****！

- ***\*直接/间接存储转发死锁\****：节点A给B发了一堆包，把B的缓冲区塞满了；B也要给A发包，把A塞满了。结果大家都发不出去，彻底卡死。
- ***\*终极解法\****：***\*结构化缓冲池 (Structured Buffer Pool)\****。根据消息走过的跳数（hops）来分配特定的缓冲区，防止环形死锁。

# 🚀 操作系统期末冲刺绝密讲义：第十五章 安全性

## 第一模块：认清底线与敌人（安全要求与四大威胁）

任何安全系统的设计，都必须死守“四大底线”，而黑客的攻击刚好与这四条底线一一对应（***\*必背配对考点\****）：

1. ***\*机密性 (Confidentiality) vs 侦听 (Interception)\****
   - **要求**：信息只能给授权的人看。
   - **威胁**：窃听、违法复制文件。这属于***\*被动攻击\****，只偷看，不搞破坏。
2. ***\*完整性 (Integrity) vs 更改 (Modification)\****
   - **要求**：资产只能由授权的人修改。
   - **威胁**：篡改数据、修改网络消息。这属于***\*主动攻击\****。
3. ***\*可用性 (Availability) vs 中断 (Interruption)\****
   - **要求**：系统必须随时能用。
   - **威胁**：破坏硬件、切断线路。或者发动***\*拒绝服务 (Denial of service, DoS)\****，用垃圾消息把网络塞满，让合法用户进不来。
4. ***\*可靠性/真实性 (Authenticity) vs 伪造 (Fabrication)\****
   - **要求**：验证你的身份到底是不是你本人。
   - **威胁**：伪装成别人 (Masquerade)、在网络中插入虚假消息。

> ***\*🌟\**** ***\*讲师防坑提示（网络攻击分类）\****：
> 考试常问“被动攻击”和“主动攻击”的区别。记住：***\*被动攻击（如通信分析、窃听）不影响系统资源，极难被发现；主动攻击（如重放、伪装、拒绝服务）会改变系统状态，必须被检测和防御\****。

## 第二模块：城防系统构建（访问控制与口令）

### 守大门：面向用户的认证（口令策略）

操作系统是怎么验证你身份的？最常见的就是 ID 和 口令（密码）。为了防黑客猜密码，系统有三大策略（***\*优缺点必考\****）：

- ***\*计算机生成口令\****：极其安全，但***\*人类根本记不住\****，容易写在纸上被偷。
- ***\*反应性口令检查 (Reactive)\****：系统定期扮演黑客去破解用户的密码，猜中了就强制用户改。缺点是***\*非常浪费资源\****。
- ***\*前摄性口令检查 (Proactive)\****：***\*目前最好的方案\****。在用户刚设置密码的时候就进行检查，不符合复杂度的直接拒绝。

### 守房间：面向数据的控制（访问矩阵）

当用户进了系统，怎么限制他能访问哪些文件？操作系统维护了一个**“访问矩阵” (Access Matrix)**，包含主体（用户）、对象（文件）和访问权（读/写）。

因为矩阵太大而且很多空格，实际系统中会把它拆开成两种形式（***\*核心对比题\****）：

- ***\*访问控制列表 (ACL - Access Control List)\****：***\*按“列”拆分\****。每个文件门上挂个牌子，写着“张三可以读，李四可以写”。
- ***\*能力票据 (Capability Tickets)\****：***\*按“行”拆分\****。给每个用户发一串钥匙，张三的钥匙串上写着“能开1号门和3号门”。

## 第三模块：黑客武器图鉴（恶意程序分类大全）

这是期末考试***\*最爱出分类填空和连线题\****的地方。恶意程序分为两大流派：

### 流派一：需要“宿主程序”（不能独立存在）

1. ***\*后门 (Trapdoor/Backdoor)\****：程序员测试时留下的秘密入口，能绕过验证。
2. ***\*逻辑炸弹 (Logic bomb)\****：像定时炸弹一样，平时没反应，一旦满足特定条件（比如到了某一天、某个文件被删），就会“爆炸”破坏系统。
3. ***\*特洛伊木马 (Trojan horse)\****：表面上是一个有用的程序（比如游戏），背地里偷偷干坏事（比如帮你把所有文件权限公开）。
4. ***\*病毒 (Viruses)\****：最狡猾！它会把自己的代码**“感染”（复制）**到其他可执行程序中。

### 流派二：独立运行的程序

1. ***\*蠕虫 (Worm)\****：***\*不需要别人运行，它自己长了脚！\**** 会通过网络连接、电子邮件主动寻找漏洞，把自己从一台机器复制到另一台机器。
2. ***\*僵尸 (Zombie)\****：秘密接管你的电脑，让你的电脑变成受黑客控制的“肉鸡”，用来对别人发动攻击，极其难以追踪。

> ***\*🌟\**** ***\*高频考点：病毒的生命周期与特殊变种\****
> ***\*四个阶段\****：潜伏 $\rightarrow$ 繁殖 $\rightarrow$ 引发 $\rightarrow$ 执行。
> ***\*多态病毒 (Polymorphic)\****：这是最难杀的病毒。它每次感染都会***\*变异并改变自己的加密密钥\****，导致传统的“特征码扫描”对它完全无效。
> ***\*宏病毒 (Macro Viruses)\****：跨平台，不感染可执行程序，***\*专挑文档（如 Word 文件）感染\****，传播极其容易。

## 第四模块：终极防御体系（入侵检测与可信系统）

### 入侵检测系统 (IDS)

操作系统怎么抓内鬼和黑客？靠观察行为。

- ***\*统计异常检测\****：收集合法用户平时的行为画像。如果你平时只看 1 个文件，今天突然下载了 1 万个，系统就判定你是异常。
- ***\*基于规则的检测\****：用专家系统直接定义什么是可疑行为。

### 反病毒方法 (Antivirus)

杀毒软件的终极武器之一是***\*通用解密 (GD)\****，它会在系统里建一个***\*CPU仿真器\****（就像一个虚拟的隔离病房），把可疑程序放进去模拟运行，看看它有没有病毒特征，从而防止真实系统被破坏。

### 🌟 最难理解的考点：可信系统 (Trusted Systems) 的多级安全

军事和金融系统需要极高的保密级别（绝密 $\rightarrow$ 机密 $\rightarrow$ 秘密 $\rightarrow$ 无密）。为了防泄密，系统通过“访问监视器”制定了两条铁律：

- ***\*不向上读 (No read up)\****：你级别低，不能看高级别的机密文件（防偷窥）。
- ***\*不向下写 (No write down)\****：***\*【最易错点】你级别高，绝对不允许\****把文件写到低级别的区域去（防止高级特工把绝密文件拷给普通级别的内鬼）！
